地域データ追加手順

このページは、新しい市区町村や地域の町丁境界データをこのツールへ追加するための手順書です。現在の構成は静的分割ロード型なので、地域ごとのデータファイルと設定を追加し、main へマージすると利用できるようになります。

今の運用では API サーバーは不要です。まずはこの手順で静的データを追加し、地域数や更新頻度が増えてきた段階でサーバー化を検討してください。

追加時に触る場所

場所 役割
data/ 地域ごとの境界データを置く場所です。最終的には .geojson と .js を配置します。
js/town-boundary-datasets.js どの住所にどのデータファイルを対応させるかを定義します。
tools/ GeoJSON をこのツール向けの軽量形式へ変換するスクリプトを置く場所です。

基本手順

  1. 元データを用意します。前提は EPSG:4326 の GeoJSON です。
  2. 地域用の変換スクリプトを作るか、既存スクリプトを複製して地域名に合わせて調整します。
  3. 変換後データを data/ に出力します。
  4. js/town-boundary-datasets.js に地域定義を追加します。
  5. ブラウザで対象地域の住所検索を行い、境界一致と半径計算を確認します。
  6. コミットしてプルリクを作成し、main にマージします。

1. 変換スクリプトを用意する

現在は tools/build-hiroshima-boundaries.mjs がひな形になります。新地域向けに複製して、出力するグローバル変数名やデータ名を地域に合わせて変更してください。

node tools/build-hiroshima-boundaries.mjs input.geojson data/hiroshima-city-boundaries.geojson

このスクリプトは次の 2 つを出力します。

  • data/xxx-boundaries.geojson: 加工済み GeoJSON
  • data/xxx-boundaries.js: ブラウザから動的ロードするための JS ファイル
新地域を増やすときは、ファイル名とグローバル変数名が衝突しないようにしてください。広島市の例なら HIROSHIMA_CITY_BOUNDARIES です。

2. data フォルダへ追加する

出力ファイルを data/ へ置きます。命名は地域が分かる形に揃えると後から見やすいです。

data/
  hiroshima-city-boundaries.geojson
  hiroshima-city-boundaries.js
  sapporo-city-boundaries.geojson
  sapporo-city-boundaries.js

3. 地域定義を登録する

js/town-boundary-datasets.js に 1 件追加します。ローダーはこの定義を見て、住所検索結果に一致するデータだけ読み込みます。

{
    key: "hiroshima-city",
    label: "広島県 広島市",
    prefecture: "広島県",
    localityAliases: ["広島市"],
    adminLevel2Aliases: ["広島市"],
    scriptPath: "data/hiroshima-city-boundaries.js",
    globalName: "HIROSHIMA_CITY_BOUNDARIES"
}

各項目の意味は次の通りです。

  • key: 内部識別子です。重複させません。
  • label: ログや画面表示向けの名前です。
  • prefecture: ジオコード結果の都道府県名と一致させます。
  • localityAliases: 通常は市区町村名を入れます。
  • adminLevel2Aliases: 必要に応じて郡や代替表記を入れます。
  • scriptPath: 動的ロードする JS ファイルの相対パスです。
  • globalName: データ JS が window に登録する変数名です。

4. 動作確認する

  1. 追加した地域の住所で検索します。
  2. ログ欄に地域データ読込の表示が出ることを確認します。
  3. 町丁境界が一致したときに半径計算へ反映されることを確認します。
  4. その地域外でも既存地域が壊れていないことを確認します。
うまく動かない場合は、都道府県名や市区町村名の表記ゆれ、globalName と生成 JS の変数名不一致をまず疑ってください。

5. 反映の考え方

今の運用では、地域データの追加はコード更新と同じ扱いです。つまり、プルリクをマージして公開環境へデプロイされれば、その時点で新地域が使えるようになります。

  • 地域データを追加する
  • 定義ファイルを更新する
  • プルリクを作成する
  • main にマージする
  • 公開環境へ反映する

将来サーバー化を考える目安

次のどれかが強くなってきたら、API サーバーやデータ管理基盤を検討する価値があります。

  • 地域追加の頻度が高く、都度デプロイが重い
  • データ量が大きくなり、さらに細かい検索や管理が必要になる
  • 非開発者でもデータ更新したくなる
  • 配信キャッシュや更新履歴を厳密に管理したい

しばらくは静的分割ロードのままで十分です。まずはこの運用で地域を増やし、痛みが出た段階でサーバー化すると無理がありません。
